# 內容研究寫作助手

此技能作為您的寫作夥伴，幫助您研究、大綱、草擬和完善內容，同時保持您獨特的聲音和風格。

## 何時使用此技能

- 撰寫部落格文章、文章或電子報
- 創建教育內容或教程
- 起草思想領袖文章
- 研究和撰寫案例研究
- 製作帶有來源的技術文件
- 撰寫適當的引用和參考資料
- 改進開場和引言
- 在寫作時獲得逐節反饋

## 此技能的功能

1. **協作大綱**：幫助您將想法結構化為連貫的大綱
2. **研究協助**：查找相關資訊並添加引用
3. **改進開場**：加強您的開場以吸引注意力
4. **分節反饋**：在您寫作時審查每個部分
5. **保留聲音**：維持您的寫作風格和語氣
6. **引用管理**：適當地添加和格式化參考資料
7. **迭代完善**：通過多次草稿幫助您改進

## 使用方法

### 設置您的寫作環境

為您的文章創建一個專用資料夾：
```
mkdir ~/writing/my-article-title
cd ~/writing/my-article-title
```

創建您的草稿檔案：
```
touch article-draft.md
```

從此目錄打開 Claude Code 並開始寫作。

### 基本工作流程

1. **從大綱開始**：
```
幫我創建一個關於 [主題] 的文章大綱
```

2. **研究並添加引用**：
```
研究 [特定主題] 並將引用添加到我的大綱中
```

3. **改進開場**：
```
這是我的引言。幫我讓開場更引人注目。
```

4. **獲取分節反饋**：
```
我剛完成了「為何重要」部分。審查它並給出反饋。
```

5. **完善和潤色**：
```
審查完整草稿的流暢性、清晰度和一致性。
```

## 指示

當使用者請求寫作協助時：

1. **了解寫作專案**

   詢問澄清問題：
   - 主題和主要論點是什麼？
   - 目標受眾是誰？
   - 所需的長度/格式是什麼？
   - 您的目標是什麼？（教育、說服、娛樂、解釋）
   - 有任何現有研究或來源要包含嗎？
   - 您的寫作風格是什麼？（正式、對話、技術）

2. **協作大綱**

   幫助結構化內容：

   ```markdown
   # 文章大綱：[標題]

   ## 開場
   - [開場句/故事/統計數據]
   - [讀者為何應該關心]

   ## 引言
   - 背景和脈絡
   - 問題陳述
   - 本文涵蓋的內容

   ## 主要部分

   ### 部分 1：[標題]
   - 關鍵點 A
   - 關鍵點 B
   - 範例/證據
   - [需要研究：特定主題]

   ### 部分 2：[標題]
   - 關鍵點 C
   - 關鍵點 D
   - 需要數據/引用

   ### 部分 3：[標題]
   - 關鍵點 E
   - 反駁論點
   - 解決方案

   ## 結論
   - 主要觀點摘要
   - 行動呼籲
   - 最終想法

   ## 研究待辦事項
   - [ ] 查找關於 [主題] 的數據
   - [ ] 獲取 [概念] 的範例
   - [ ] 為 [主張] 尋找引用來源
   ```

   **迭代大綱**：
   - 根據反饋調整
   - 確保邏輯流暢
   - 識別研究空白
   - 標記需要深入研究的部分

3. **進行研究**

   當使用者請求研究某個主題時：

   - 搜索相關資訊
   - 查找可信來源
   - 提取關鍵事實、引用和數據
   - 以請求的格式添加引用

   範例輸出：
   ```markdown
   ## 研究：AI 對生產力的影響

   主要發現：

   1. **生產力提升**：研究顯示內容創建任務節省 40% 時間 [1]

   2. **採用率**：67% 的知識工作者每週使用 AI 工具 [2]

   3. **專家引用**："AI 增強而非取代人類創造力" - Dr. Jane Smith, MIT [3]

   引用：
   [1] McKinsey Global Institute. (2024). "The Economic Potential
       of Generative AI"
   [2] Stack Overflow Developer Survey (2024)
   [3] Smith, J. (2024). MIT Technology Review interview

   已添加到大綱的第 2 部分。
   ```

4. **改進開場**

   當使用者分享引言時，分析並加強：

   **當前開場分析**：
   - 有效之處：[積極元素]
   - 可以更強的地方：[改進領域]
   - 情感影響：[當前 vs. 潛力]

   **建議替代方案**：

   選項 1：[大膽陳述]
   > [範例]
   *為何有效：[解釋]*

   選項 2：[個人故事]
   > [範例]
   *為何有效：[解釋]*

   選項 3：[令人驚訝的數據]
   > [範例]
   *為何有效：[解釋]*

   **開場問題**：
   - 它是否引起好奇心？
   - 它是否承諾價值？
   - 它是否足夠具體？
   - 它是否符合受眾？

5. **提供逐節反饋**

   當使用者撰寫每個部分時，審查：

   ```markdown
   # 反饋：[部分名稱]

   ## 有效之處 ✓
   - [優點 1]
   - [優點 2]
   - [優點 3]

   ## 改進建議

   ### 清晰度
   - [特定問題] → [建議修正]
   - [複雜句子] → [更簡單的替代方案]

   ### 流暢性
   - [過渡問題] → [更好的連接]
   - [段落順序] → [建議重新排序]

   ### 證據
   - [需要支持的主張] → [添加引用或範例]
   - [籠統陳述] → [使其更具體]

   ### 風格
   - [語氣不一致] → [更好地匹配您的聲音]
   - [詞彙選擇] → [更強的替代方案]

   ## 具體行編輯

   原文：
   > [草稿的確切引用]

   建議：
   > [改進版本]

   原因：[解釋]

   ## 要考慮的問題
   - [發人深省的問題 1]
   - [發人深省的問題 2]

   準備進入下一部分！
   ```

6. **保留作者的聲音**

   重要原則：

   - **學習他們的風格**：閱讀現有的寫作樣本
   - **建議，不要替換**：提供選項，而非指令
   - **匹配語氣**：正式、休閒、技術、友好
   - **尊重選擇**：如果他們更喜歡自己的版本，支持它
   - **增強，不要覆蓋**：使他們的寫作更好，而非不同

   定期詢問：
   - "這聽起來像您嗎？"
   - "這是正確的語氣嗎？"
   - "我應該更/更少 [正式/休閒/技術] 嗎？"

7. **引用管理**

   根據使用者偏好處理參考資料：

   **內嵌引用**：
   ```markdown
   研究顯示生產力提高 40% (McKinsey, 2024)。
   ```

   **編號參考**：
   ```markdown
   研究顯示生產力提高 40% [1]。

   [1] McKinsey Global Institute. (2024)...
   ```

   **腳註風格**：
   ```markdown
   研究顯示生產力提高 40%^1

   ^1: McKinsey Global Institute. (2024)...
   ```

   維護運行中的引用列表：
   ```markdown
   ## 參考資料

   1. Author. (Year). "Title". Publication.
   2. Author. (Year). "Title". Publication.
   ...
   ```

8. **最終審查和潤色**

   當草稿完成時，提供全面反饋：

   ```markdown
   # 完整草稿審查

   ## 整體評估

   **優點**：
   - [主要優點 1]
   - [主要優點 2]
   - [主要優點 3]

   **影響**：[整體有效性評估]

   ## 結構與流暢性
   - [組織評論]
   - [過渡質量]
   - [節奏評估]

   ## 內容質量
   - [論點強度]
   - [證據充分性]
   - [範例有效性]

   ## 技術質量
   - 語法和機制：[評估]
   - 一致性：[評估]
   - 引用：[完整性檢查]

   ## 可讀性
   - 清晰度分數：[評價]
   - 句子多樣性：[評價]
   - 段落長度：[評價]

   ## 最終潤色建議

   1. **引言**：[具體改進]
   2. **正文**：[具體改進]
   3. **結論**：[具體改進]
   4. **標題**：[如需要的選項]

   ## 發布前檢查清單
   - [ ] 所有主張都有來源
   - [ ] 引用格式正確
   - [ ] 範例清晰
   - [ ] 過渡流暢
   - [ ] 存在行動呼籲
   - [ ] 校對錯別字

   準備發布！
   ```

## 範例

### 範例 1：Teresa Torres 的工作流程

**使用者**："我正在寫一篇關於持續發現的文章。幫我創建一個大綱。"

**處理過程**：
1. 協作大綱結構
2. 識別研究需求
3. 使用者開始撰寫引言
4. 審查和改進開場
5. 使用者撰寫每個部分
6. 在每個部分後提供反饋
7. 進行研究並添加引用
8. 完整草稿的最終審查
9. 潤色並準備發布

**結果**：一篇研究充分、引用適當的文章，以 Teresa 的聲音撰寫，具有強大的結構和流暢性。

### 範例 2：研究密集型文章

**使用者**："我正在寫關於 AI 對產品管理的影響。幫我找到當前數據和範例。"

**輸出**：
```markdown
## 編譯的研究

### 近期數據
- Gartner：到 2025 年 80% 的 PM 將使用 AI 工具 [1]
- 調查：45% 報告更快的功能發布 [2]
- 案例研究：Airbnb 的 AI 輔助優先級排序 [3]

### 專家引用
- "AI 放大 PM 判斷，而非取代它" - Marty Cagan
- [附有引用的其他引用]

### 真實範例
1. **公司 A**：使用 AI 進行用戶研究綜合
   - 結果：節省 60% 時間
   - 來源：[引用]

2. **公司 B**：AI 驅動的路線圖分析
   - 結果：更好的利益相關者對齊
   - 來源：[引用]

所有內容已添加到您的大綱中並附有適當的引用。
```

### 範例 3：開場改進

**使用者的原始開場**：
> "產品管理因 AI 而改變。在這篇文章中，我將討論 AI 影響產品經理的一些方式。"

**改進選項**：

**選項 1（數據驅動）**：
> "上個月，我讓 AI 分析了 500 個客戶訪談。它花了 30 分鐘而不是 3 週。產品管理將永遠不同。"

**選項 2（問題）**：
> "如果您可以在早晨咖啡前與每位客戶交談、閱讀每條評論並分析每張支援票證，會怎樣？"

**選項 3（故事）**：
> "Sarah 花了兩週時間開發錯誤的功能。不是因為她不了解用戶，而是因為她無法足夠快地處理數百次訪談以發現模式。"

### 範例 4：分節反饋

**使用者**："剛完成我的'常見錯誤'部分。有什麼想法嗎？"

**回應**：
```markdown
# 反饋：常見錯誤部分

## 有效之處 ✓
- 強有力的範例（Slack 通知故事）
- 清晰的結構（3 個錯誤，乾淨的佈局）
- 每個錯誤的實用建議

## 建議

### 使其更具體
您的第二點說"糟糕的優先級排序"但可以更強：

當前：
> "許多團隊通過專注於功能而非結果來進行糟糕的優先級排序。"

建議：
> "我看過團隊在一個季度發布 14 個功能，但沒有移動任何關鍵指標。他們優先考慮活動而非進展。"

### 添加數據
第三個錯誤將受益於證據：
> "[添加引用]：研究顯示沒有定期用戶接觸的團隊構建未使用功能的可能性高 3 倍 [需要來源]"

### 流暢性改進
考慮重新排序：錯誤 3 → 錯誤 2 → 錯誤 1
這從小到大影響構建。

準備進入下一部分！
```

## 寫作工作流程

### 部落格文章工作流程
1. 一起大綱
2. 研究關鍵點
3. 撰寫引言 → 獲得反饋
4. 撰寫正文部分 → 每個部分的反饋
5. 撰寫結論 → 最終審查
6. 潤色和編輯

### 電子報工作流程
1. 討論開場想法
2. 快速大綱（較短格式）
3. 一次會議起草
4. 審查清晰度和連結
5. 快速潤色

### 技術教程工作流程
1. 大綱步驟
2. 撰寫程式碼範例
3. 添加解釋
4. 測試說明
5. 添加故障排除部分
6. 準確性的最終審查

### 思想領袖工作流程
1. 頭腦風暴獨特角度
2. 研究現有觀點
3. 發展您的論點
4. 以強烈的觀點撰寫
5. 添加支持證據
6. 撰寫引人注目的結論

## 專業技巧

1. **在 VS Code 中工作**：比 Web Claude 更適合長篇寫作
2. **一次一個部分**：逐步獲得反饋
3. **單獨保存研究**：保留 research.md 檔案
4. **版本您的草稿**：article-v1.md、article-v2.md 等
5. **大聲朗讀**：使用反饋識別笨拙的句子
6. **設定截止日期**："我想今天完成草稿"
7. **休息**：撰寫、獲得反饋、暫停、修訂

## 檔案組織

```
~/writing/article-name/
├── outline.md          # 您的大綱
├── research.md         # 所有研究和引用
├── draft-v1.md         # 第一稿
├── draft-v2.md         # 修訂稿
├── final.md            # 準備發布
├── feedback.md         # 收集的反饋
└── sources/            # 參考資料
    ├── study1.pdf
    └── article2.md
```

## 最佳實踐

### 對於研究
- 在引用之前驗證來源
- 盡可能使用近期數據
- 平衡不同觀點
- 連結到原始來源

### 對於反饋
- 具體說明您想要什麼："這太技術性了嗎？"
- 分享您的擔憂："我擔心這部分拖沓"
- 提出問題："這在邏輯上流暢嗎？"
- 請求替代方案："還有什麼其他方式來解釋這個？"

### 對於聲音
- 分享您的寫作範例
- 指定語氣偏好
- 指出良好匹配："那聽起來像我！"
- 標記不匹配："對我的風格來說太正式了"
