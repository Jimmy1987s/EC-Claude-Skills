# Jimmy Style Pro - 專業溝通風格指南

## 核心理念

**「先講清楚，再講人話」**

整合邏輯思考框架與 Jimmy Nian 風格，先用結構化思維整理內容，再用輕鬆語調潤飾。
適合策略文件、提案、報告等正式場合，但不失親和力。

## 使用時機

當使用者需要：
- 整理雜亂的策略思考或會議記錄
- 重寫正式但沉悶的商業文件
- 撰寫需要邏輯清晰又好讀的提案
- 即時溝通時希望兼顧專業與親和

## 混合模式支援

這個 skill 支援三種場景：
1. **文字內容重寫與優化** - 把冗長、正式的文字改寫成 Jimmy 風格
2. **即時溝通指導** - 在寫 email、Slack 訊息時提供風格建議
3. **文件撰寫輔助** - 從零開始撰寫結構化文件

## 三大邏輯框架

### 1. 金字塔原則（Pyramid Principle）- 預設框架

**使用時機**：策略報告、提案、執行摘要

**運作方式**：
- 先講結論（What）
- 再展開 3-5 個關鍵論點（Why）
- 每個論點下提供支持細節（How）

**輸出結構**：
```
## 核心結論
[一句話說清楚要幹嘛]

## 三大關鍵
### 1. 第一個論點
- 支持細節 A
- 支持細節 B

### 2. 第二個論點
...

## 一句話總結
> [金句收尾]
```

### 2. MECE 原則（Mutually Exclusive, Collectively Exhaustive）

**使用時機**：分類問題、資源盤點、選項分析

**運作方式**：
- 確保分類不重疊（互斥）
- 確保涵蓋所有可能（完整）
- 適合「三大命脈」「四個階段」這類分類

**輸出結構**：
```
## [主題] 完整盤點

### 類別 A：[名稱]
- 項目 1
- 項目 2

### 類別 B：[名稱]
- 項目 1
- 項目 2

（類別之間不重疊，加起來涵蓋全部）
```

### 3. Think-Feel-Do 框架

**使用時機**：說服性文件、變革管理、溝通計畫

**運作方式**：
- **Think（邏輯）**：提供數據和理性論據
- **Feel（情感）**：連結到讀者的痛點或願景
- **Do（行動）**：明確的下一步行動

**輸出結構**：
```
## 為什麼要做這件事？（Think）
[數據、事實、理性論據]

## 這對你有什麼好處？（Feel）
[痛點共鳴、願景描繪]

## 現在該怎麼做？（Do）
[具體行動步驟]
```

## 自動偵測規則

根據輸入內容的特徵，自動判斷使用哪種模式：

| 輸入特徵 | 偵測模式 | 套用框架 |
|----------|----------|----------|
| 包含「策略」「規劃」「提案」「報告」「藍圖」 | 策略模式 | 金字塔原則 |
| 包含「分類」「盤點」「列表」「項目」「類型」 | 分類模式 | MECE 原則 |
| 包含「說服」「推動」「變革」「溝通」「為什麼」 | 說服模式 | Think-Feel-Do |
| 包含「整理」「重寫」「優化」「改寫」 | 重寫模式 | 自動選擇最適合 |
| 無明顯特徵 | 通用模式 | 金字塔原則（預設） |

### 手動指定模式

使用者可以在呼叫時明確指定：
- `/jimmy-style-pro pyramid` → 強制使用金字塔原則
- `/jimmy-style-pro mece` → 強制使用 MECE 原則
- `/jimmy-style-pro tfd` → 強制使用 Think-Feel-Do

## Jimmy 風格潤飾規則

### 語調轉換表

| 正式寫法 | Jimmy 風格 |
|----------|-----------|
| 本專案旨在... | 這專案要解決的是... |
| 經評估後建議採用... | 評估後，建議採用... |
| 上述方案具有以下優勢... | 這方案的優勢在於... |
| 請於期限內完成 | 目標是月底前完成 |
| 綜上所述 | 總結來說 |
| 基於上述分析 | 從分析來看 |
| 本報告將探討 | 這報告聚焦在... |
| 經過審慎評估 | 評估後 |
| 建議採取以下措施 | 做法如下 |
| 必須立即執行 | 這需要優先處理 |
| 不可接受 | 這部分需調整 |
| 務必注意 | 關鍵點要注意 |
| 嚴禁 | 要避免 |
| 應當如此 | 正確做法是 |
| 可能會有風險 | 有個風險要留意 |
| 尚待確認 | 這點待確認 |

### 結構性元素保留

這些邏輯結構**不能因為潤飾而被破壞**：
- 金字塔的層次關係
- MECE 的分類邊界
- 數據和事實的準確性
- 專有名詞和技術術語

### 適度幽默的邊界

- ✅ 可以加：小括號補充、適量 emoji 點綴、口語化比喻
- ✅ 可以加：「簡單說就是」「白話文版本」這類過渡語
- ❌ 不要加：影響專業性的玩笑、過多 emoji、網路流行語
- ❌ 不要加：降低可信度的自嘲、過於隨便的語氣

### 格式規範

```markdown
## 大標題（用動詞或問句）

### 小標題（具體描述）

**粗體**：強調關鍵結論

- 條列式：細節說明
- 每條不超過兩行

> 引言區塊：一句話總結或金句

---（分隔線用於區分大段落）
```

## 輸出範例

### 輸入（原始雜亂文字）
```
核心產業知識技術部分要把他們都規格化收集起來，AI協助指引和規格化。
萬向插頭部分的資料庫規格這個月底前我和資訊理完他，主要是大命脈的
1. ERP和2. CRM 資料清理，第三段就是市場訊息的爬蟲和關鍵資訊分析...
```

### 輸出（Jimmy Style Pro）
```markdown
# 資料庫整合與 AI 應用開發藍圖

## 核心結論

把散落各處的產業知識標準化收進資料庫，讓 AI 可以讀得懂、用得上。

## 三大命脈資料清理

### 1. ERP 資料清理
把 ERP 裡的髒資料整乾淨，該標準化的標準化。

### 2. CRM 資料清理
客戶資料也要整理，不能再亂七八糟了。

### 3. 市場訊息爬蟲 + 關鍵資訊分析
- 爬市場上的公開訊息
- 抓出對決策有影響的關鍵資訊
- 依照「能改變效益的關鍵決策」來設計資料庫結構

## 一句話總結

> 先把 ERP/CRM/市場資料理乾淨收進 AWS，開好萬向接口讓前端透過 API 讀取。
```

## 與 jimmy-nian-style 的差異

| 特性 | jimmy-nian-style（原版） | jimmy-style-pro（本 skill） |
|------|-------------------------|----------------------------|
| 主要用途 | 技術文件、README、註解 | 商業文件、策略報告、提案 |
| 語調 | 輕鬆幽默為主 | 專業嚴謹，適度幽默 |
| 結構 | 較隨性 | 邏輯框架主導 |
| 目標讀者 | 開發者 | 管理層、商業決策者 |
| 邏輯框架 | 無 | 金字塔/MECE/Think-Feel-Do |

---

## 注意事項

- 技術準確性 > 幽默感（永遠不犧牲正確性）
- 保持結構完整性（邏輯框架是骨架，不能散掉）
- 考慮讀者層級（給高層看要更精簡，給執行層看要更細節）
- 數據和事實不加油添醋

## 一句話總結

> 用邏輯框架把內容理清楚，再用 Jimmy 風格讓人願意讀下去。專業但不無聊，這就是 Jimmy Style Pro。
